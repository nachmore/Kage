# Internationalisation (i18n)

Kage ships with built-in localisation across all three layers — Rust backend,
frontend, and extension sandbox — so users can run in their own language and
extension authors can translate their own strings without writing any
infrastructure.

This document covers:

1. The architecture of the i18n system.
2. How to add a new string.
3. How to add a new language.
4. How translations are kept in sync (drift-check CI gate).
5. Right-to-left (RTL) language handling.
6. Extension authoring with i18n.
7. Why logs stay English.

For day-to-day work the short version is:

- Rust code uses `t!("key.path", "name" => value)`.
- JS code uses `t("key.path", { name: value })`.
- Every key lives in `locales/en/messages.json` (canonical, hand-authored).
- `python scripts/translate.py` fills in the other 30 languages via Claude.
- `python scripts/check_i18n.py` is a CI gate; missing keys fail the build.

## 1. Architecture

### Catalog format

One `messages.json` file per language under `locales/<code>/`. Entries are
keyed by a dotted path:

```json
{
  "_meta": {
    "language": "en",
    "name": "English",
    "rtl": false,
    "machine_translated": false
  },
  "errors.connection.lost": {
    "message": "Connection lost: {reason}",
    "description": "ACP transport dropped while a request was in flight"
  },
  "chats.count": {
    "message": "{count, plural, one {1 chat} other {# chats}}"
  }
}
```

Placeholders use ICU MessageFormat syntax. The supported subset is:

- `{name}` — simple substitution.
- `{count, plural, one {…} other {…}}` — CLDR plural categories. Supports
  `=N` exact-match arms, `#` for the count value, and nested `{name}`.
- `{role, select, admin {…} other {…}}` — discrete switch.

Categories use `Intl.PluralRules` on the JS side and a manual category match
on the Rust side (which is conservative — Rust callers should keep templates
simple and let the frontend do plural-heavy rendering).

The `description` field is translator-facing context and is consumed only by
`scripts/translate.py`. The `_machine_translated` flag (per entry, in non-EN
catalogs) marks AI-generated translations so a future human review pass can
sweep them.

### Where catalogs are stored

- **Rust** — every catalog is embedded into the binary at compile time via
  `include_str!` in `src/i18n.rs`. No runtime disk I/O, no resource path
  drama. Adding a language requires adding a line to the `embed_locales!`
  macro invocation.
- **Frontend** — the host calls `get_i18n_catalog` (a Tauri command in
  `src/commands/i18n.rs`) on window startup. It returns the active locale's
  catalog plus an English fallback. The result is cached in `ui/js/shared/i18n.js`.
- **Extensions** — each extension ships its own catalog at
  `_locales/<code>/messages.json` (Chrome WebExtensions convention). The
  extension manager calls `read_extension_locale` (or fetches via HTTP for
  bundled extensions) and hands the result to the sandbox at init time.
  The runtime exposes `context.i18n.t(key, vars)` to extension code.

### Language resolution

1. `config.ui.language` (explicit user override, set in Settings → Appearance).
2. `sys-locale::get_locale()` (OS locale via the platform-native API).
3. Fall back to `en`.

Region tags strip down: `en-GB → en`, `zh-CN → zh-CN` (kept) → `zh` (next
fallback). The frontend defers to whatever Rust resolves; there's a single
source of truth for the active language.

When the user changes the language in Settings, the backend persists
`config.ui.language` and broadcasts `config_updated`. Each window's
`i18n.js` listens for this, re-fetches the catalog, applies static
translations, sets `<html dir>`, and dispatches `kage:i18n-changed` so
modules can re-render their dynamic content.

### Where translation happens

**Translation happens once, at the boundary, not in business logic.** Logs
stay English. Domain code carries i18n keys plus parameters; the strings
materialise only at the surface that's about to render them.

For Rust errors specifically:

```rust
return Err(AppError::keyed(
    ErrorKind::ConnectionLost,
    "errors.connection.lost",
    &[("reason", &e.to_string())],
));
```

`AppError` carries `(kind, key, params)`. Its `Display` impl renders against
the *English* catalog so `log::error!("{}", e)` produces stable, greppable
output regardless of the user's UI locale. The frontend gets the message
materialised in the active locale because Tauri serialises the error via
the custom `Serialize` impl, which calls `i18n::translate(active_locale, ...)`
at that one boundary.

Free-form upstream errors (agent text, JSON-RPC payloads we don't control)
go through `AppError::raw(kind, message)`, which routes the inner text
through the translatable `errors.passthrough` template. The framing is
localised; the inner text is verbatim.

## 2. Adding a new string

### Backend (Rust)

1. Add the key to `locales/en/messages.json` with a `message` and a
   translator-facing `description`.
2. Use `t!("key.path", "name" => value, ...)` from the call site:
   ```rust
   let label = t!("tray.show");
   let err = AppError::keyed(ErrorKind::Internal, "errors.foo.bar", &[("path", &p)]);
   ```
   Param values must be `&str` — call `.to_string()` or `&format!(...)` for
   numbers / Display impls.
3. Run `python scripts/translate.py` to populate the other 30 languages.

### Frontend (JS)

1. Add the key to `locales/en/messages.json`.
2. Use `t("key.path", { name: value })`:
   ```js
   import { t } from '../shared/i18n.js';
   button.textContent = t('common.save');
   const greeting = t('chat.greeting', { name: user.name });
   ```
3. For static HTML, use `data-i18n="key.path"` (or `data-i18n-title`,
   `-placeholder`, `-aria-label`, `-alt`, `-html`) and call
   `applyStaticTranslations(root)` after the markup is in the DOM.
4. Plurals:
   ```js
   t('chat.unread_count', { count: 7 })
   // catalog: "{count, plural, one {1 unread chat} other {# unread chats}}"
   ```

### Extensions

1. Create `_locales/en/messages.json` in the extension directory.
2. Use `__MSG_key__` tokens in the manifest's `name` and `description`:
   ```json
   {
     "id": "demo",
     "name": "__MSG_manifest.name__",
     "description": "__MSG_manifest.description__"
   }
   ```
3. In provider code, use the `i18n` proxy from the context:
   ```js
   initialize(context) {
     this.t = context.i18n.t;
   }
   match(query) {
     return [{
       label: this.t('result.label'),
       description: this.t('result.description'),
     }];
   }
   ```
4. Re-run `python scripts/translate.py` (this script also picks up extension
   catalogs).

## 3. Adding a new language

The starter set covers the CLDR top 30 (English plus 30 other tongues). To
add a new one:

1. Add the language code + display name + RTL flag to
   `DEFAULT_LANGS` in `scripts/translate.py`.
2. Add an `embed!` line in `src/i18n.rs` so the binary embeds the catalog
   (requires a recompile).
3. Run `python scripts/translate.py --langs <code>` to seed translations
   from the canonical EN catalog. The script calls Claude with the EN
   message + description + target language and writes the result back
   to `locales/<code>/messages.json`.
4. Run `python scripts/check_i18n.py` to verify drift-check passes.
5. The Settings → Appearance language dropdown picks up the new entry
   automatically (the list is fetched from
   `get_available_languages` at runtime).

For a region-tagged variant (e.g. `zh-TW` alongside `zh-CN`) the same flow
applies — the resolver falls back from `zh-HK` → `zh` → `en` if no
exact match ships.

## 4. Drift-check CI gate

`scripts/check_i18n.py` runs as part of `python scripts/test_all.py`. It
fails the build when any of:

- A `t!(...)` / `t(...)` call references a key not in `locales/en/messages.json`
  (typo or forgotten catalog entry).
- A non-English catalog has a different set of keys than EN.
- A non-English translation drops or adds a `{placeholder}` relative to EN
  (would cause `{name}` to render literal at runtime, or runtime crashes
  when the translator added an unknown placeholder).
- An extension is missing `_locales/en/messages.json`.
- Any extension's locale catalog drifts from its EN source.

The CI gate is hard-fail on every drift mode (per project policy: we use AI
to fill in gaps, so any drift is fixable in one re-run of `translate.py`).
Stale EN keys are reported as **warnings**, not errors — they won't break a
build but should be cleaned up.

The drift-check tool itself has self-tests at `scripts/test_check_i18n.py`.
Both run from `python scripts/test_all.py`.

### Limitations

- The checker scans for **static** keys only (string literals). Dynamic
  keys (Rust `t!(some_var)`, JS template strings) are silently skipped.
  If you must use a dynamic key, list the resolved set in
  `KNOWN_DYNAMIC_KEYS` at the top of `scripts/check_i18n.py`.
- Test code is excluded from the scan (test fixtures use `t()` in
  assertions and shouldn't gate the build).

## 5. Right-to-left (RTL) handling

Active languages flagged `rtl: true` cause:

- `<html dir="rtl">` and `<html lang="ar">` (or whatever code).
- `body.classList.add('rtl')`.
- CSS rules in `ui/css/shared-rtl.css` flip the layout-affecting properties
  via `[dir=rtl]` selectors.

The CSS approach is selector-based rather than logical-property-based on
purpose: we ship modern WebView2 / WebKit, but `[dir=rtl]` rules are the
easier-to-grep choice for a UI that already has a small CSS surface.

**Per-input RTL detection** (the existing `ui/js/shared/rtl.js`) is
orthogonal: it adds a `.rtl` class to a textarea container when the user
types Arabic into an LTR UI. The two systems coexist:

- `shared-rtl.css` = whole-app direction (window chrome).
- `rtl.js` = per-input direction (typed content).

**Anything that should stay LTR** inside an RTL page (URLs, code blocks,
hotkey display, monospace IDs) should carry `dir="ltr"` or
`class="ltr"` — the CSS file has a `.ltr` rule for this.

## 6. Why logs stay English

User-visible text gets translated; everything else doesn't:

- `log::info!` / `warn!` / `error!` / `debug!` always emit English.
- `println!` / `eprintln!` always emit English.
- `console.log` / `warn` / `error` always emit English.
- `Display for AppError` (used by `format!("{}", e)` in logs) emits English
  even when the active locale is something else.

Reasoning: `app.jsonl` from a Japanese user's machine should be searchable
from any developer's environment. If a user reports "I see error 'コード3'",
the developer can't grep for that in the codebase. With English-only logs,
the user's report and the source code share a vocabulary.

The serialised payload to the frontend is the only place the active locale
materialises into a string. From that point onward (the UI), everything is
in the user's language.

## 7. Glossary of files

| File                                       | Purpose |
|--------------------------------------------|---------|
| `locales/<code>/messages.json`             | Per-language catalog. EN is canonical. |
| `src/i18n.rs`                              | Rust loader, `t!` macro, `translate()`, `set_language()`. |
| `src/error.rs`                             | `AppError` carries `(kind, key, params)`; serialises with active locale. |
| `src/commands/i18n.rs`                     | `get_i18n_catalog`, `get_available_languages`, `set_language`. |
| `ui/js/shared/i18n.js`                     | Frontend `t()`, ICU subset, static-markup applier. |
| `ui/js/extension-sandbox/runtime.js`       | Extension `context.i18n.t()` proxy. |
| `ui/js/shared/extension-manager.js`        | Loads `_locales/<lang>/messages.json` per extension. |
| `ui/css/shared-rtl.css`                    | RTL CSS rules. Loaded in every window. |
| `scripts/check_i18n.py`                    | Drift-check CI gate. |
| `scripts/test_check_i18n.py`               | Self-tests for the drift checker. |
| `scripts/translate.py`                     | Claude-powered seed / update of non-EN catalogs. |
| `Kage-Extensions/extensions/<id>/_locales/`| Per-extension catalogs. Same shape. |

## 8. Common pitfalls

- **Don't pre-translate strings in domain code.** Don't `format!("Failed to {}: {}", verb_translated, err)` — instead build a key for the whole template.
- **Don't translate constants used as wire format.** Event names, command names, IDs, file extensions, agent presets — those go on the wire and stay English.
- **Don't add `_machine_translated: true` per entry by hand.** That flag is a contract between `translate.py` and the human reviewer who eventually QAs the entry; if you remove it, `translate.py` will leave the entry alone on subsequent runs.
- **Don't forget plurals.** `"{count} chats"` is wrong even in English (it
  says "1 chats" for `count=1`). Use the plural form.
- **Don't use `t()` for log lines.** Logs are English, every time, by design.
- **Always validate placeholders match the EN source.** A translation that
  drops `{reason}` will render the literal string `{reason}` at runtime
  unless the drift-check catches it (which it does — but don't ship a
  bypass).
